## How It Works

OpenDST acts as a puppet master for the JVM. When your code calls a non-deterministic JDK API, OpenDST's agent redirects it to the `Simulator`.

- **Virtual Time** — Time only moves when tasks are executed. No more real-world waits.
- **Deterministic Randomness** — Automatically seeded from the simulation plan for 100% reproducibility.
- **Single-Threaded Loop** — Virtual Threads are scheduled on a discrete-event loop to explore interleavings.
- **Simulated Networking** — Virtual IP stacks with programmable partitions, latency, and connection resets.
- **State-Space Assertions** — The orchestrator uses `sometimes` and `always` assertions to guide branching.

## Built for the Modern JVM

OpenDST isn't just a wrapper. It's a low-level engineering feat that leverages the latest advancements in the Java ecosystem to provide guaranteed determinism.

1. **ClassFile API (JEP 457)** — Utilizes the new JDK 25 ClassFile API for high-performance bytecode manipulation and offline instrumentation.
2. **Virtual Threads (JEP 444)** — Forcing all concurrency through Virtual Threads managed by the simulator's deterministic scheduler.
3. **Two-Tier Instrumentation** — Combines offline Maven transformation with a runtime Java Agent for full JDK coverage.
4. **Session-Wide Liveness** — Verifies that properties eventually become true across all simulation runs in a session.

## Current Limitations

:::caution[Experimental: Evolving rapidly. Expect breaking changes.]
:::

:::caution[No `Thread` Inheritance]
Classes extending `Thread` are not supported. Use `Runnable` or Virtual Threads instead.
:::

:::caution[Blocking JNI]
Carrier threads are pinned during JNI calls. Blocking JNI methods will halt the simulator until they return.
:::

:::caution[Java 25+ Required]
Requires the latest JVM features (ClassFile API, JEP 491) for instrumentation and scheduling.
:::

:::caution[Strict Node Isolation]
All workload logic must run within virtual nodes defined via the Deployment API to be correctly intercepted.
:::
