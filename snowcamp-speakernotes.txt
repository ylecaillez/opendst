Slide suivant:
Si on y réfléchit, la manière dont nous ecrvons habituellement les tests, le
example-based testing, ne permet pas de tirer profit d’une des plus grandes
resources dont nous disposons: la puissance de calcul phénoménale des machines
modernes.
Imaginons que je fasse une modificaiton non trivial d’une partie du code de mon
application. J’éxécute la suite de test complete qui peut prendre plusieurs heures. A la
fin j’ai un résultat a peut prés vert avec deux ou trois
Flaky. Je relance pour être sur. Les flaky on disparu,, un nouveau est apparu mais il
était green avant. Job done.
Mais entre ces deux runs, les tests effectués était exactement les mêmes (a la
flakiness pret). Je n’ai pas tester plus intensement mes modifications en lançant ces
tests deux fois …
Par contre si on execute un generateurs de tests, là l’histoire est totalement
différente. Chaque run va générer de nouveaux tests qui vont tester des nouveaux
cas et potentiellement tester d’autre partie du code et pourquoi pas
trouver un bug ! Puisque c’est de ça dont il s’agit. C’est une course entre notre
capacité a trouver les bugs et la capacité de notre environement de prod a trouver
ces mêmes bugs.
Du coup si on écrit un générateurs de test on passe un peu moins de temps de
développement que d’écrire tout une suite qui va essayer de tester tous les cas 
auquel on passe. Et on va aussi passer beaucoup moins de temps à la maintenir.
On échange du temps de développement qui coute cher avec du temps machine qui
coute moins cher. En plus d’être efficace, c’est très rentable.
D’ailleur a priori c’est ce que fait google avec ClusterFuzz. Ils ont 30k VM qui tournent
h24 pour faire ça. Je suis convaincu que ce n’est pas les seuls.

Slide suivant:
Parceque cette idée de générer des tests au runtime n’est pas nouvelle. Cette idée
date de 1988 et c’est l’idée de Barton Miller a l’University of Wisconsin en 1988
A l’origine l’intention c’était de tester un CLI Unix, pour ça ils ont écrit un générateur
de test qui a l’executoin aller appelr le CLI a chaque fois avec des argument généré
aléatoirement.
Si l’application crash, bravo, vous avez trouvé un bug.
Cette technique a ensuite été adopté pour tester les protocoles réseau, les librairie de
gestion de format de fichier etc … super pratique pour détecter des buffer overflow
etc.
Pour tester un protocol réseau, on imagine bien qu’on allait pas partir avec une
approche du type “testQueLeProtocolPeutEnvoyerLaLettreA”,
“testQueLeRéseauMarcheQuandJ’envoieLaLettreB”,
“testQueLeRéseauMarcheQuandJ’envoieLaLettreApuisB”, …
Avec le fuzz testing on écrit un générateur de test qui va générer des dixaine voir des
centaines de milliers de tests.
Des dixaine ou des centaines de milleirs de tests PAR SECONDE.
Soudain notre garçon de restaurant dans l’example de tout à l’heure va être bien
occupé.

rencontrer. Et cet espace est immense.

La propriété de symétrie est géniale pour tester tout ce qui est
sérialization/désérialization. Avant j’écrivais des tests du style testCanSerializePojo,
testCanSerializePojoWithAnArray, test …
Maintenant j’écris un property-based test qui
1. Generate random POJO graph
2. Serialize it
3. Deserialize it
4. Compare with the original
5. Profit

pour faire du code coverage donnera pas mal d’info. Notamment les parties de code
jamais exécutés.
